## 프로젝트 개요

### 목적
다국어 협력사 데이터(44,313행)에서 중복 업체를 탐지하고 그룹번호를 부여하는 시스템

### 핵심 원칙
- ✅ **중복 탐지**: 같은 업체로 판단되는 행들을 찾아서 그룹번호 부여
- ✅ **원본 데이터 유지**: 모든 원본 행 유지 (병합하지 않음)
- ✅ **다국어 지원**: 같은 업체의 다른 언어 표기도 탐지
- ✅ **2단계 구조**: STEP1(후보 판단) + STEP2(최종 판단)

### 데이터 특성
- **데이터 크기**: 약 44,313행, 22열
- **지원 언어**: 중국어, 영어, 한글, 일본어, 포르투갈어, 베트남어, 인도네시아어, 태국어 등
- **핵심 문제**: 같은 업체가 다른 언어로 표기된 경우도 중복으로 탐지해야 함

## 요구사항

### 1. 다국어 중복 탐지 ⭐ 핵심
- 같은 업체의 다른 언어 표기도 중복으로 탐지
- 예시:
  - "阿里巴巴集团" (중국어) ↔ "Alibaba Group" (영어) → 같은 업체
  - "삼성전자" (한글) ↔ "Samsung Electronics" (영어) → 같은 업체
  - "株式会社ABC" (일본어) ↔ "ABC Corporation" (영어) → 같은 업체

### 2. 출력 형식
- 그룹번호를 맨 앞 컬럼으로 추가
- `그룹번호_STEP1`: 후보 그룹 번호
- `그룹번호_STEP2`: 최종 중복 그룹 번호
- `_duplicate_count`: 중복 그룹의 행 수
- 모든 원본 행 유지 (병합하지 않음)

### 3. 성능 요구사항
- 빠른 처리 (44k 행을 효율적으로 처리)
- 정확한 중복 탐지 (오매칭 최소화)

### 4. 중간 결과 저장
- 5개 그룹 발견 시 자동으로 중간 결과 파일 저장
- 최종 결과 파일과 동일한 형식

## 핵심 기능

### 1. 다국어 중복 탐지
- **로마자화**: 중국어→Pinyin, 일본어→Romaji, 한글→로마자화
- **의미 기반 임베딩**: `paraphrase-multilingual-MiniLM-L12-v2` 모델 사용
- **다중 Blocking 키**: 원본 키 + 로마자화 키 + 영어 변형 키 등

### 2. 하이브리드 방식
- **Blocking**: 비교 횟수 대폭 감소 (151M → 수만~수십만)
- **의미 기반 임베딩**: 정확도 최대화 (92-96%)
- **RapidFuzz**: 빠른 문자열 비교

### 3. 오매칭 방지
- 업종 키워드 비교 (다른 업종 업체 간 오매칭 방지)
- 법인 형태 충돌 확인 (중국어 특화)
- 일반 단어 블랙리스트 (인도네시아어 등)
- 주소 정확도 강화 (건물/번지 단위까지 확인)

## 개발 과정 및 히스토리

### 1단계: 초기 설계 및 구현
- **문제**: 기존 방식은 언어별로 Blocking 키가 분리되어 다국어 중복을 탐지하지 못함
- **해결**: 다중 Blocking 키 생성 + 의미 기반 임베딩

### 2단계: 사용자 피드백 반영
- 출력 형식 변경 (그룹번호 추가, 원본 행 유지)
- 오매칭 사례 분석 및 개선
- 이름 유사도 기준 조정 (0.7)
- 중간 결과 저장 기능 추가

### 3단계: 성능 최적화
- 국가 전체 와일드카드 키 제거 (큰 그룹 생성 방지)
- 큰 그룹 임계값 낮춤 (100 → 50)
- 큰 그룹 비교 횟수 감소 (100개 → 50개)
- **결과**: 전체 예상 시간 대폭 단축 (큰 그룹 크기 감소로 인한 성능 향상)

### 4단계: 정확도 개선 (GPT 피드백 반영)
- 더 엄격한 중복 판단 기준 적용
- 법인명 핵심어 추출 및 비교 강화
- 건물/번지 단위 주소 일치 확인
- 업종 키워드 비교 추가

### 5단계: 오매칭 사례 분석 및 수정
- 포르투갈어 업종 키워드 추가
- 번지 번호 차이 확인 강화
- 핵심어 추출 개선 (공백 유지)
- 공통 prefix 보너스 추가

### 6단계: 2단계 구조 도입 (GPT 프롬프트 기반)
- **STEP1: 중복 후보(CANDIDATE) 여부 판단** (Recall 중심)
  - `is_candidate()` 함수 구현
  - 느슨한 기준으로 가능성이 있으면 후보로 분류
  - 고유 단어 일부 공통, 이름 유사도 ≥0.70, 같은 국가/도시 등
- **STEP2: 동일 법인(SAME_ENTITY) 최종 판단** (Precision 중심)
  - `are_duplicates()` 함수 개선
  - 엄격한 기준으로 후보 중에서만 최종 판단
  - 고유 상호 일치 필수, 주소 일관성, 업종 충돌 없음 등
- `detect_duplicates()` 함수 수정: 2단계 구조로 변경, 튜플 반환 `(candidate_groups, final_groups)`
- `merge_duplicates_2step()` 함수 추가: STEP1, STEP2 컬럼 생성
- 결과 파일에 `그룹번호_STEP1`, `그룹번호_STEP2` 컬럼 추가

## 중복 판단 기준

### GPT 프롬프트 기반 2단계 구조

#### STEP 1: 중복 후보(CANDIDATE) 여부 판단 (Recall 중심)
다음 중 하나라도 만족하면 후보로 판단:
1. 업체명에서 법인형태·지역명·일반단어를 제거했을 때 의미 있는 고유 단어가 일부라도 겹침
2. 약어, 띄어쓰기, 하이픈, 번역 차이로 인해 동일 상호의 표기 변형일 가능성
3. 같은 국가이고, 주소의 도시/성(Province) 단위가 동일함
4. 과거 데이터 정합성 관점에서 "같은 업체로 등록되었을 가능성"을 배제할 수 없음

#### STEP 2: 동일 법인(SAME_ENTITY) 최종 판단 (Precision 중심)
다음 조건을 모두 만족하는 경우에만 SAME_ENTITY로 판단:

**1. 고유 상호(Core Business Name) 비교**
- 고유 상호가 실질적으로 동일해야 함
- 철자, 띄어쓰기, 구두점, 단어 순서 차이는 허용
- 의미가 다른 단어 조합은 허용되지 않음

**2. 주소 비교 (보조 지표)**
- 주소가 거의 동일하면 동일 업체 가능성을 강화
- 같은 도시라도 상세 주소가 다르면 보조 근거로만 사용
- 주소만 같고 업체명이 다르면 동일 업체로 판단하지 않음
- 서로 다른 도시/성에 위치하면 기본적으로 다른 업체

**3. 업종 충돌 확인 (강제 차단)**
- 업종 키워드가 명확히 다르면 동일 업체로 판단하지 않음
- 예: 건설 vs 제조, 광고 vs 보일러, 특수강 vs 밸브

**4. 법인 형태 충돌 확인 (중국어 특화)**
- 법인 형태가 다르면 다른 업체로 판단
- 예: "建设工程总公司" (건설 공사 회사) vs "股份有限公司" (주식회사) → 다른 업체

**5. 일반 단어 블랙리스트**
- 다음 단어들은 고유 상호로 절대 간주하지 않음:
  - 중국: 商贸, 物资, 集团, 制造厂, 建设工程
  - 인도네시아: SEJAHTERA, MAKMUR, ABADI, JAYA, MANDIRI, CENTRAL
  - 베트남: CTY, CONG TY, TNHH, SX, TM, DV, BAO BI
  - 영어/포르투갈어: TRADING, COMMERCE, INDUSTRIAL, SOLUTION, INTEGRATION

### 최종 판단 기준
다음 조건을 모두 만족하는 경우에만 SAME_ENTITY로 판단:
1. **고유 상호가 실질적으로 동일함** (필수 조건)
   - 고유 상호가 일치하지 않으면 즉시 False 반환
   - 일반 단어를 제외한 고유 단어 공통 비율이 30% 미만이면 False
2. **주소 일관성** (보조 지표)
   - 공급업체코드 일치 또는 건물/번지 일치 또는 주소 유사도가 충분히 높음
   - 주소만 같고 업체명이 다르면 동일 업체로 판단하지 않음
3. **업종 충돌이 없음**
   - 업종 키워드가 명확히 다르면 False 반환
   - 같은 브랜드명이 있으면 업종 충돌 완화
4. **법인 형태 충돌이 없음** (중국어의 경우)
   - 법인 형태가 다르면 False 반환
   - 특히 "建设工程总公司"는 다른 법인 형태와 절대 같을 수 없음

**핵심 원칙**: 고유 상호 일치가 필수이고, 주소는 보조 지표로만 사용됨. 확실하지 않으면 같은 업체로 판단하지 않음 (False Positive 방지)

그 외 모든 경우는 DIFFERENT_ENTITY로 판단

## 2단계 구조 (GPT 프롬프트 기반)

### 구조 개요
프로젝트는 GPT 프롬프트에 따라 2단계 구조로 재구성되었습니다:

1. **STEP 1 (후보 판단)**: Recall 중심 - 가능성이 있으면 후보로 분류
2. **STEP 2 (최종 판단)**: Precision 중심 - 후보 중에서만 엄격한 기준으로 최종 판단

### 구현 세부사항

#### `is_candidate()` 함수 (STEP 1)
- 목표: 같은 업체일 가능성이 있어 추가 검토가 필요한 쌍인지 판단
- 기준: 느슨한 기준 (고유 단어 일부 공통, 이름 유사도 ≥0.70, 같은 국가/도시 등)
- 출력: `(is_candidate: bool, reason: str)`

#### `are_duplicates()` 함수 (STEP 2)
- 목표: 실제로 병합해도 되는 동일 법인인지 판단
- 기준: 엄격한 기준 (고유 상호 일치, 주소 일관성, 업종 충돌 없음 등)
- 출력: `(is_duplicate: bool, confidence: float)`

#### `detect_duplicates()` 함수
- STEP 1: 모든 쌍에 대해 `is_candidate()` 호출 → 후보 그룹 생성
- STEP 2: 후보인 경우에만 `are_duplicates()` 호출 → 최종 중복 그룹 생성
- 반환값: `(candidate_groups: List[List[int]], final_groups: List[List[int]])`

### 결과 파일 구조
- `그룹번호_STEP1`: 후보 그룹 번호 (추가 검토 필요)
- `그룹번호_STEP2`: 최종 중복 그룹 번호 (실제 중복)
- `_duplicate_count`: STEP2 최종 그룹의 행 수

## 오매칭 사례 분석 및 해결

### 사례 1: "COMERCIAL ALFERES PIRACICABA LTDA." vs "UNIODONTO PIRACICABA"
**문제**: 다른 업종인데 같은 업체로 판단
**원인**: 포르투갈어 업종 키워드 미인식
**해결**: 포르투갈어 업종 키워드 추가
- '의료': 'UNIODONTO', 'ODONTOLOGIA', 'DENTAL', 'DENTISTRY'
- '무역': 'COMERCIAL', 'COMERCIO', 'COMMERCIAL'

### 사례 2: 번지 차이 무시
**문제**: 번지가 다른데 (1665 vs 1352) 같은 업체로 판단
**원인**: 건물/번지 일치 로직이 번지 번호 차이를 제대로 확인하지 않음
**해결**: 번지 번호가 다르면 False 반환

### 사례 3: "3DB CONSULTORIA LTDA ME" vs "3DB SOLUTIONS LTDA ME"
**문제**: 같은 업체인데 다른 업체로 판단
**원인**: 
- 핵심어 추출 과정에서 "CONSULTORIA"가 "NSULTORIA"로 변형됨
- 주소 유사도가 낮게 계산됨
**해결**:
- 핵심어 추출 시 공백 유지 (단어 간 구분 보존)
- 공통 prefix 보너스 추가
- 건물/번지 매칭에 숫자 매칭 보너스 추가

### 사례 4: SEJAHTERA 케이스 (인도네시아)
**문제**: "ALMEGA SEJAHTERA", "ANUGRAH TANI SEJAHTERA" 등이 같은 업체로 묶임
**원인**: "SEJAHTERA"는 인도네시아에서 일반적인 수식어로, 고유 상호가 아님
**해결**: 인도네시아 일반 단어 블랙리스트 추가
- SEJAHTERA, MAKMUR, ABADI, JAYA, SENTOSA, MANDIRI

### 사례 5: 그룹번호 2, 3, 5 (일반 단어만 공통)
**문제**: "CENTRA NU" vs "CENTRAL PART", "CENTRAL NIAGA" vs "CENTRAL TEKNINDO PERKA" 등이 같은 업체로 묶임
**원인**: 일반 단어(INDONESIA, CENTRAL, SOLUSI, INTEGRASI)만 공통으로 있어도 매칭됨
**해결**: 일반 단어를 제외한 고유 단어만 비교
- 고유 단어 공통 비율이 30% 미만이면 다른 업체로 판단

### 사례 6: "沈阳化工建设工程总公司" vs "沈阳化工股份有限公司"
**문제**: 법인 형태가 다른데 같은 업체로 판단
**원인**: 법인 형태 차이를 구분하는 로직 없음
**해결**: 법인 형태 충돌 확인 로직 추가
- "建设工程总公司" (건설 공사 회사) vs "股份有限公司" (주식회사) → 다른 업체

## 결과 확인 가이드

### 결과 파일 구조

#### 기본 컬럼 (원본 데이터)
- `공급업체코드`: 공급업체 코드
- `공급업체명`: 공급업체 이름
- `Land`: 국가 코드
- `CITY1`, `CITY2`: 도시 정보
- `STREET`: 거리 정보
- `HOUSE_NUM1`, `HOUSE_NUM2`: 건물 번호
- `STR_SUPPL1`, `STR_SUPPL2`, `STR_SUPPL3`: 추가 주소 정보
- 기타 원본 데이터의 모든 컬럼

#### 추가된 컬럼 (중복 탐지 정보)
- `그룹번호_STEP1`: STEP1 후보 그룹 번호 (추가 검토 필요)
  - 같은 후보 그룹 번호를 가진 행들은 중복 가능성이 있음
  - STEP2에서 최종 판단되지 않은 후보 그룹도 포함
- `그룹번호_STEP2`: STEP2 최종 중복 그룹 번호 (실제 중복)
  - 같은 최종 그룹 번호를 가진 행들은 같은 업체로 판단됨
  - 중복이 아닌 행은 None
- `_duplicate_count`: STEP2 최종 그룹의 행 수 (중복이 아닌 행은 None)
  - 해당 그룹에 몇 개의 중복 행이 있는지 표시

### 결과 확인 방법

#### Excel에서 확인하기
1. Excel에서 파일 열기
2. `그룹번호_STEP2` 컬럼으로 정렬 (내림차순)
3. 값이 있는 행들이 최종 중복 그룹입니다
4. `그룹번호_STEP1` 컬럼으로 필터링하여 후보 그룹 확인 가능

#### Python으로 확인하기
```python
import pandas as pd

# 결과 파일 로드
df_result = pd.read_excel('bio_vendor_merged_hybrid_YYYYMMDD_HHMMSS.xlsx')

# STEP2 최종 중복 그룹만 보기
final_duplicates = df_result[df_result['그룹번호_STEP2'].notna()]
print(f"최종 중복 행 수: {len(final_duplicates)}개")

# STEP1 후보 그룹만 보기
candidates = df_result[df_result['그룹번호_STEP1'].notna()]
print(f"후보 그룹 행 수: {len(candidates)}개")

# 가장 많이 중복된 상위 10개
top_merged = final_duplicates.nlargest(10, '_duplicate_count')
print("\n가장 많이 중복된 상위 10개:")
for idx, row in top_merged.iterrows():
    print(f"\n{row['공급업체명']} (Land: {row['Land']})")
    print(f"  중복 행 수: {row['_duplicate_count']}개")
```

### 검증 방법
1. **병합 정확도 확인**: `그룹번호_STEP2`가 같은 행들이 실제로 같은 업체인지 확인
2. **누락 확인**: 원본 파일의 총 행 수와 결과 파일의 총 행 수 비교 (모든 행이 유지되어야 함)
3. **데이터 품질 확인**: 병합된 행에서 중요한 필드가 올바르게 유지되었는지 확인

## 핵심 알고리즘 및 로직

### 1. Blocking 키 생성 (`create_multiple_blocking_keys`)
**키 형식**: `{국가}|{이름prefix}|{도시명}`

**생성되는 키**:
- 원본 이름 prefix + 원본 도시명
- 로마자화 이름 prefix + 로마자화 도시명
- 영어 단어별 prefix + 도시명
- 국가만 있는 경우: `{국가}||`

**주의**: 국가 전체 와일드카드 키(`{country}|*|*`)는 제거됨 (너무 큰 그룹 생성 방지)

### 2. 고유 상호 추출 (`extract_company_core_name`)
**불용어 제거**:
- 중국어: 有限公司, 股份有限公司, 公司, 企业 등
- 영어: LTD, LIMITED, INC, CORP, COMPANY 등
- 한글: 주식회사, 유한회사, 법인 등
- 포르투갈어: LTDA, ME, EIRELI, SA 등
- 베트남어: CONG TY, TNHH 등
- 인도네시아어: PT, CV, TB, UD 등
- 일반 단어: SEJAHTERA, MAKMUR, ABADI, JAYA, MANDIRI, CENTRAL 등

**처리 과정**:
1. 악센트 제거
2. 불용어 제거 (긴 단어부터)
3. 공백 정리 (단어 간 구분 유지)
4. "CO"는 맨 끝에 있을 때만 제거 (예: "IND.ENG.CO"는 유지)

### 3. 건물/번지 일치 확인 (`check_building_address_match`)
**비교 필드**:
- `STREET` (도시명이 아닌 경우만)
- `HOUSE_NUM1`, `HOUSE_NUM2`
- `STR_SUPPL1`, `STR_SUPPL2`, `STR_SUPPL3`

**로직**:
- 번지 번호가 다르면 False 반환
- 숫자 매칭 보너스 추가 (공통 숫자가 있으면 보너스 점수)
- 숫자 매칭 보너스가 있으면 기준 완화 (0.85 → 0.75)

### 4. 주소 유사도 계산 (`calculate_address_similarity`)
**계산 방식**:
- `ratio` (30%) + `partial_ratio` (50%) + `token_sort_ratio` (20%)
- 공통 단어/숫자 보너스 추가
- 도시명만 같을 때는 점수 제한 (건물/번지 정보가 없으면 낮은 점수)

### 5. 업종 키워드 비교 (`check_business_type_match`)
**지원 업종**:
- 건축, 전력, 화학, 전자, 기계, 의료, 운송, 무역, IT, 식품, 조명, 자동차

**다국어 키워드**:
- 한글, 영어, 일본어, 중국어, 포르투갈어, 베트남어 지원
- 다른 업종이면 중복이 아님
- 같은 브랜드명이 있으면 업종 충돌 완화
